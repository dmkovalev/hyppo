# Дизайн GUI для hyppo (React + FastAPI)

Дата: 2026-07-06. Статус: на ревью.

## Назначение и аудитория

GUI — часть публичного open-source hyppo, ориентирован на **внешних исследователей** (не на демо защиты и не только на автора). Приоритеты: onboarding нового пользователя, наглядность жизненного цикла виртуального эксперимента (ВЭ), простота установки. Существующий модуль `hyppo/streamlit` (заглушка) объявляется устаревшим и заменяется этим GUI.

Первая версия покрывает **полный цикл**: определение ВЭ → граф гипотез → план → выполнение → результаты/сравнение → повторное использование в следующей итерации.

## Дистрибуция

Локальный запуск по модели Jupyter/MLflow:

```
pip install hyppo
hyppo gui        # поднимает FastAPI на localhost:8787, открывает браузер
```

Никакой внешней инфраструктуры. Многопользовательский серверный режим и десктоп (Tauri) — вне рамок этой версии; архитектура их не блокирует (фронтенд отделён от ядра HTTP-границей).

## Архитектура

```
hyppo/
├── hyppo/gui/            # новый пакет: FastAPI-приложение
│   ├── app.py            # создание приложения, раздача статики webui
│   ├── api/              # REST-роутеры: projects, hypotheses, graph,
│   │                     #   plans, runs, comparison
│   ├── ws.py             # WebSocket: прогресс выполнения
│   └── services.py       # тонкий слой над manager/planner/runner/
│                         #   metadata_repository (ядро не меняется)
└── webui/                # Vite + React + TypeScript
    ├── src/routes/       # экраны (см. ниже)
    ├── src/components/   # LifecycleRing, IterationHistory, RunsTable, ...
    └── dist/             # собранная статика, пакуется в wheel
```

Принципы:
- **GUI stateless**: всё состояние — в существующем `metadata_repository`; перезапуск сервера ничего не теряет.
- **Ядро не трогаем**: `services.py` только вызывает публичные методы модулей ядра; новые возможности ядра — отдельные задачи вне этого дизайна.
- REST для CRUD и чтения; WebSocket только для живого прогресса выполнения.
- Библиотеки фронтенда: React Flow (граф гипотез), Plotly.js (parallel coordinates, графики метрик). Без Next.js/SSR.

## Навигация

Сайдбар: список проектов; внутри проекта разделы: **Обзор, Гипотезы, Граф, Планы, Запуски, Сравнение**.

### Обзор (главная проекта) — утверждённый «гибрид»

1. **Кольцо-индикатор** (компактное): текущая итерация, этап N из 6, краткий статус («Выполнение: 12 из 24, осталось ~40 мин»), подсказка следующего шага. Метафора цикла, а не линейного конвейера.
2. **История итераций** (паттерн DVC Studio / GitLab CI): каждая итерация — строка, новые сверху; в строке — статус, мини-пайплайн этапов, дельта к предыдущей итерации (лучшая гипотеза, ΔR², доля переиспользованного кэша, число отсечённых маршрутов). Каскадное переиспользование результатов — ключевое отличие hyppo — видно именно здесь.
3. **Контекстные действия**: одна-две крупные кнопки, зависящие от этапа («Смотреть прогресс», «Перейти к сравнению»).

Отвергнутые альтернативы: канвас-центричный интерфейс (плохо масштабируется на большие решётки), линейная лента этапов (читается как конвейер с концом, теряет итеративность), чистый workspace без обзора (нет целостной картины цикла).

## Экраны по этапам цикла

| Экран | Содержимое | Ядро hyppo |
|---|---|---|
| Гипотезы | Мастер определения ВЭ: онтология, гипотезы, параметры и домены Q_i; автоматический расчёт размера пространства конфигураций и допустимого подпространства; редактирование после создания | `manager`, `ontology` |
| Граф | React Flow: узлы-гипотезы, цвет = эпистемический статус (proposed / supported / refuted / superseded); рёбра derived_by / competes; виртуализация при больших графах (опыт Dagster) | `lattice_constructor`, `coa` |
| Планы | Разбиение P_ne / P_e (пересчёт vs кэш), бюджет, пороги отсечения; предпросмотр «что будет пересчитано, если изменить X» | `planner` |
| Запуски | Таймлайн запусков (полоса = запуск, длина = длительность) + таблица: выполняется / завершён / ошибка / отсечён (с причиной); живой прогресс по WebSocket | `runner` |
| Сравнение | Таблица метрик по гипотезам + parallel coordinates (параметры → метрика, паттерн W&B); выбор итераций для сравнения | `comparison`, `metadata_repository` |

## Демо-проект norne-brugge

Предустановлен, виден при первом запуске. Данные — подмножество из наборов Norne/Brugge (уже используются в experiments/chapter4). Назначение: onboarding «клик по демо → пройти цикл, ничего не настраивая» и живой пример для документации. Размер урезается до быстрого прогона (минуты, не часы).

## Поток данных и ошибки

- Клиент опрашивает REST для списков/деталей; при активном выполнении подписывается на WS-канал итерации.
- Упавший запуск не прерывает итерацию: строка получает статус «ошибка» + лог; итерация завершается частично, история это отражает.
- Разрыв WS — переподключение с восстановлением состояния из REST (единственный источник истины — metadata_repository).
- Валидация определения ВЭ (корректная определённость) выполняется ядром; GUI показывает нарушенные условия списком с привязкой к полям мастера.

## Тестирование

- **API**: pytest на роутеры поверх временного metadata_repository (существующая тестовая инфраструктура, ~259 тестов рядом).
- **UI**: Playwright smoke на критический путь: создать проект → определить 2–3 гипотезы → построить граф → запустить → увидеть результат и вторую итерацию с кэшем. Прогон на демо norne-brugge.
- Компонентные тесты — только для LifecycleRing и IterationHistory (нетривиальная логика отображения дельт).

## Вне рамок первой версии

Многопользовательский режим и аккаунты; Tauri-десктоп; редактирование онтологии визуальным редактором; генерация гипотез из данных через GUI (вызов есть, конструктора нет); мобильная вёрстка.
