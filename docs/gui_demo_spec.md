# Спецификация GUI-демонстраций: алгоритмы 1–4, Теорема 1, сложности, правило 5

Назначение: воспроизвести в GUI все проверяемые утверждения статей живыми
вызовами платформы. **Источник истины — `tests/test_golden_claims.py`**:
каждая демонстрация обязана сходиться с одноимённым golden-тестом. Ничего
не переопределять локально — только реальные вызовы hyppo.

Обозначения гипотез: в коде — исторические имена (`H11…H15`, `H12b`, `GRP`,
`H19`), в статье СВД — сплошная нумерация H1–H16. Соответствие:
`H11→H9, H12→H10, H12b→H11, H13→H12, H14→H13, H15→H14, GRP→H15, H19→H16`.
В GUI показывать нумерацию статьи, имена кода — в tooltip.

## Демо 1. Алгоритм 1 — автоматическое построение графа гипотез

```python
from hyppo.lattice_constructor._base import HypothesisLattice
from hyppo.coa._base import Equation, Structure
# 16 формул NORNE_HYPS — скопировать из tests/test_golden_claims.py
G = HypothesisLattice(hypotheses, workflow).lattice   # networkx.DiGraph
```

**Golden**: ровно 18 рёбер `NORNE_GOLDEN_EDGES` (см. тест
`test_alg1_norne_graph_matches_figure`), 16 узлов, DAG, глубина 10.

**Визуализация**: граф (две ветви + слияние, как рис. 3 статьи). На ребре —
объяснение вывода: «выход `I_agg` (H1) входит в уравнение H2». Выход
гипотезы = левая часть её формулы (fix `5907005` — не менять!).

## Демо 2. Алгоритм 2 — инкрементальное добавление

```python
lat = HypothesisLattice(hypotheses[:-1], workflow)   # без H19
lat.add_hypothesis(h19)                              # инкрементально
```

**Golden**: граф после добавления == полной перестройке на всех 16
(`test_alg2_incremental_equals_full_rebuild`); новые рёбра: `H8→H19`,
`H15→H19`.

**Визуализация**: кнопка «добавить гипотезу» → появляются и подсвечиваются
только новые рёбра; счётчик каузальных объединений показывает |H| (а не |H|²).

## Демо 3. Алгоритм 3 — двухэтапная проверка корректной определённости

```python
from hyppo.ontology.consistency import check_consistency, Status
res = check_consistency(ve, onto, lattice, run_hermit=..., artefacts=..., configurations=...)
```

Стадия B (структурная, `run_hermit=False`, мгновенно):

| Сценарий | Вход | Ожидаемый статус |
|---|---|---|
| корректный | DAG + согласованные артефакты + конечные домены | `OK` |
| цикл | `{0:{1},1:{2},2:{0}}` | `C3` + свидетель цикла |
| разрыв потока | ребро с `Out(i)∩In(j)=∅` | `C4` + виновное ребро |
| бесконечный домен | объект без `finite=True` | `C5` |
| двойное нарушение | цикл + разрыв | `C3` (порядок проверок фиксирован) |

Стадия A (семантическая, HermiT, ~секунды — показывать спиннер):

- **обнаружимо**: гипотеза с двумя разными моделями при функциональном
  свойстве + AllDifferent → «онтология противоречива» (C2);
- **ключевая демонстрация OWA**: гипотеза *без* модели — рассуждатель
  говорит «согласовано»! Показать плашку: «открытый мир не видит
  отсутствия — эту проверку выполняет маркерный слой (правило 9)».
  Это исполняемое обоснование трёхслойной архитектуры (тест
  `test_alg3_stage_a_owa_cannot_see_missing_model`).

Стадию A запускать в изолированном `owlready2.World()` — не трогать общую
онтологию приложения.

## Демо 4. Алгоритм 4 — план пересчёта + каскад

```python
from hyppo.coa.graph import HypothesisGraph
g = HypothesisGraph.from_edges(n, edges)   # топология из Демо 1
pne = g.plan(cached)                       # множество к пересчёту
```

**Golden**: `plan(cached)` == «некэшированные + все достижимые из них»
(`test_alg4_plan_matches_reachability_oracle`).

**Визуализация**: пользователь кликами отмечает кэшированные узлы →
подсветка P_ne. Сценарий статьи: инвалидация H1 красит всю ветвь жидкости
(H2–H5, H8) и H16, ветвь обводнённости не тронута; ГТМ (H15) красит только
H5, H8, H16.

## Демо 5. Теорема 1 — минимальность плана

Интерактив «предложи свой план»: пользователь выбирает произвольное
множество P. GUI проверяет два условия корректности
(`_is_valid_plan` из golden-тестов): P содержит все некэшированные; P
замкнуто по потомкам (каскадность A2). Затем показывает:

- P некорректно → указать нарушенное условие;
- P корректно → подсветить, что `plan(cached) ⊆ P`, вывод: «ваш план
  содержит минимальный; лишние вершины: P \ plan». Минимальность и по
  числу вершин, и по любой неотрицательной стоимости.

**Golden**: `test_theorem1_plan_is_minimal_correct_plan` (на графах n≤6 —
честный перебор всех подмножеств; в GUI перебор показывать только для
малых графов).

## Демо 6. Сложности — операционными счётчиками, не секундами

Инструментирование: обернуть `hyppo.coa.causal.is_complete` счётчиком
(monkeypatch на время демо), для Алг. 4 — счётчик обходов смежности.

| Алгоритм | Вход (цепочка длины n) | Счётчик | Закон |
|---|---|---|---|
| Алг. 1 `build()` | n = 10/20/40 | проверок полноты | ровно n(n−1)/2 → ×4 при ×2 |
| Алг. 2 `add_hypothesis` | n = 10/20/40 | объединений | ровно n → ×2 при ×2 |
| Алг. 4 `plan()` | n = 200/400/800 | обходов смежности | ~V+E → ×2 при ×2 |

**Визуализация**: столбики «счётчик vs теоретическая кривая» с подписями
O(|H|²·s·v) / O(|H|·s·v) / O(V+E). Не использовать время выполнения —
только счётчики (детерминированно).

## Демо 7. Правило 5 — процедурная ацикличность

```python
from hyppo.ontology.consistency import _find_cycle_via_kahn
```

Попытка добавить ребро, замыкающее цикл, → отказ с показом свидетеля
(последовательность вершин цикла). Плашка: «транзитивное свойство в OWL
нельзя объявить асимметричным (ограничение simplicity) — ацикличность
проверяется процедурно при каждом добавлении связи» [СВД §3.2].

## Демо 8. Серия из 80 испытаний (систематическая проверка обнаружения)

Источник данных: **`norne_battery_results.json`** (в корне репозитория) —
результаты реальных 80 запусков HermiT, сгенерированные `norne_battery.py`.
НЕ запускать серию живьём в интерфейсе (247 с) — грузить готовый JSON;
опционально кнопка «пересчитать» с прогресс-баром, вызывающая скрипт в фоне.

Структура JSON: `rule4[]` — 16 записей `{source, stale[], match}`;
`rule7[]` — 64 записи `{run_uses, invalid, derived_stale, expected, verdict}`;
`summary` — итоги.

**Визуализация**:
- правило 4 — таблица «источник → число устаревших» (монотонно убывает
  от 12 у H1 до 0 у прогноза) с зелёными галочками совпадения с оракулом;
  при наведении подсвечивать соответствующий каскад на графе демо 1;
- правило 7 — матрица 4×16 (запуск × кандидат), клетки TP/TN, ошибок нет;
- итоговая плашка: «102 классификации и 64 пары — 0 расхождений
  с достижимостью в графе; проверка соответствия реализации спецификации,
  а не статистическая метрика» [СВД, §Систематическая проверка].

**Golden**: `summary` обязан давать rule4_classifications=102,
rule4_mismatches=0, rule7_tp=43, rule7_tn=21, rule7_errors=0.
